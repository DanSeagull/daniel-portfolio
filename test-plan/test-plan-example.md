# Тест-план: Мобильное приложение "Яндекс Go" (версия для пассажиров)

## 1. Объект тестирования
**Название продукта:** Яндекс Go — суперапп для городских сервисов (такси, каршеринг, доставка, аренда самокатов).

**Платформы:** iOS (последние 2 версии), Android (последние 3 версии).

**Ключевой функционал (Scope):**
- **Основной сценарий:** Вызов такси "сейчас" (Выбор точки А → Выбор точки Б → Подбор машины → Отслеживание в реальном времени → Оплата → Оценка поездки).
- **Платежи:** Безналичная оплата (карта/ApplePay/GooglePay), наличный расчет, промокоды.
- **AI-функционал (новое):** Чат с "Алисой AI" для вызова такси голосом или текстом.
- **Личный кабинет:** История поездок, настройки профиля, способы оплаты.
- **Безопасность:** Проверка номера телефона (SMS), соответствие машины/водителя в приложении.

*Не тестируем в этом плане:* Нагрузочное тестирование бэкенда, тестирование даркмода, тестирование версии для водителей/курьеров.

---

## 2. Виды тестирования

| Вид тестирования | Почему это важно для Яндекс Go | Инструменты |
|-------------------|-------------------------------|--------------|
| **Функциональное** | Пользователь должен доехать от А до Б без сбоев. | Ручное, Чек-листы |
| **UI/UX** | Карта — главный экран. Важно отображение машин, зума, такси на дороге. | Figma, DevTools (Remote Debugging) |
| **Интеграционное** | Приложение зависит от 3х вещей: Карты (GPS), Платежный шлюз, SMS-провайдер. | Charles Proxy (сниффер) |
| **Тестирование локации (GPS)** | Сердце приложения. Как ведет себя при слабом сигнале, в метро, под мостами. | Charles (Mocking локации), Android Studio Emulator |
| **Тестирование AI (Алиса)** | Проверка корректности распознавания голосовых команд (шум, акценты) и текстовых интентов. | Логи мобильного приложения, ручное тестирование |
| **Регрессионное** | Каждое обновление карты или Алисы не должно сломать оплату. | Автотесты (UI-тесты на критический путь) |
| **Кросс-платформенное** | Разное поведение кнопки "Назад" на Android и свайпов на iOS. | Xcode Simulator, Android Studio |

---

## 3. Подход к тестированию

**Ручное тестирование:**
- **Исследовательское:** Имитация реальных поездок в разных районах города (центр, спальник, промзона).
- **Негативное:** Попытка вызвать такси без интернета, с нулевым балансом на карте, отмена поездки на разных этапах.

**Инструменты для эмуляции (Важно для портфолио):**
- **Charles / Proxyman:** Для подмены ответов сервера (например, заставить приложение думать, что рядом нет машин).
- **Android Studio / Xcode:** Для эмуляции слабого GPS-сигнала или зоны с плохой связью (Network Link Conditioner).
- **Postman:** Для проверки API-ручек (например, получение списка доступных тарифов).

---

## 4. Критерии качества (Exit Criteria)

Мы завершаем тестирование, когда:
1.  **P0 (Блокеры):** Нет багов, мешающих оформить заказ или оплатить поездку.
2.  **Точность геолокации:** Ошибка определения адреса "А" не превышает 50 метров.
3.  **Алиса:** Алиса распознает команду "Домой" и строит маршрут в 95% случаев.
4.  **Память:** Приложение не вылетает (нет Crashlytics ошибок) при фоновой работе в течении 10 минут.

## 5. Ресурсы и тестовые данные

**Тестовое окружение:**
- **Устройства:** iPhone 15 (iOS 17), Pixel 6 (Android 14), эмулятор iPhone SE (маленький экран).
- **Аккаунты:**
    - Пользователь с привязанной картой Visa.
    - Пользователь с привязанной картой Mastercard (с 3D Secure).
    - Пользователь с нулевым балансом.
    - Пользователь с активным промокодом ("TEST2025").
- **Геоданные:** Тестовая локация (Москва, ул. Тверская).

## 6. Риски (Риски тестирования такси)

| Риск | Вероятность | Митигация (Что сделаю я) |
|------|-------------|--------------------------|
| **Финансовые риски** (списание лишних денег) | Средняя | Акцент на тестах API оплаты и проверке итоговой суммы в истории. |
| **"Сюрприз" от AI** (Алиса отправила не туда) | Высокая | Покрыть тестами не только логичный ввод, но и сленг ("Мчу", "Газуй", "Шевелись"). |
| **Проблемы с реальным GPS** (в помещении) | Низкая | Использовать эмуляцию "плохого сигнала" в Charles/Android Studio. |

---

## 7. Расписание (Пример на 1 спринт)

- **День 1:** Анализ требований к новому тарифу "Детский" + подготовка чек-листов.
- **День 2:** Функциональное тестирование (ручное) + Запись багов в баг-трекинг (имитируем Jira, Яндек Трекер).
- **День 3:** Тестирование API и Интеграций (Postman, Charles).
- **День 4:** Регресс (проверка, что не сломался старый функционал оплаты).
- **День 5:** Составление отчета и демо.
